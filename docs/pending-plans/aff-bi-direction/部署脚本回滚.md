部署脚本、迁移机制、迁移清单我都核对过了。先给你一个关键结论,它会大幅简化你的回滚预案,然后给出完整的操作顺序。

  核心结论:这次的迁移几乎全是"增量",代码回滚通常不需要还原数据库

  PR#39 带了 9 个迁移(165–173)。我逐个看了它们动什么:

  - 165、166、173:全是 CREATE TABLE(新表:user_recharge_discounts、referal_reward_tracker、referral_spend_events、activity_*)。
  - 167、168、169、170、171:都只动这批新表(改表的约束、加新表的列/索引),旧代码根本不认识这些表。
  - 172:唯一一个碰已有表的——给 user_affiliates 加了一个可空列 inviter_bound_at,并回填。可空、纯增量。

  这意味着一个非常重要的事实:旧二进制(当前 main)可以直接跑在已迁移的新库上而不出错。因为没有任何旧代码用到的东西被删除或改成不兼容的形态——新表它当没看见,新的可空列它也忽略。

  再加上迁移 runner 的机制(我看了 migrations_runner.go):

  - 只向前,没有 down/rollback;按文件名 + SHA256 记录在 schema_migrations。
  - runner 只遍历"二进制里嵌入的迁移",逐个对照数据库。schema_migrations 里多出来的、旧二进制不认识的行会被直接忽略,不会报错。

  所以回滚分两个层级,绝大多数情况只需第一层:

  ---
  部署前准备(按顺序)

  先开维护窗口,然后:

  # 1. 记录当前部署的代码版本(回滚参照,比 pre-deploy 分支更精确)
  git -C "$HOME/app/sub2api" rev-parse HEAD | tee ~/deploy_prev_commit.txt

  # 2. 停服务(进入静默,保证 DB 备份一致 + 防止窗口内产生会被回滚丢掉的数据)
  sudo systemctl stop sub2api

  # 3. 备份正在运行的二进制 ★关键★
  #    注意:manual_update.sh 会先 rm 掉 /opt/sub2api/sub2api 再 build,
  #    若 build 失败旧二进制就没了。必须事先备份。
  sudo cp /opt/sub2api/sub2api /opt/sub2api/sub2api.bak

  # 4. 备份配置(迁移不动它,但一起备了稳妥)
  sudo cp /opt/sub2api/config.yaml /opt/sub2api/config.yaml.bak

  # 5. 备份数据库(服务已停,快照完全一致)
  mkdir -p ~/backups
  PGPASSWORD=sub2api pg_dump -h localhost -U sub2api -d sub2api \
    -Fc -f ~/backups/sub2api_$(date +%Y%m%d_%H%M%S).dump

  你说的"先停服务、再备份数据库"——对,顺序正确。pg_dump 本身是事务一致快照,但停服务能保证窗口内没有新写入,这样万一回滚也不会丢用户数据。

  ---
  部署

  迁移是在新二进制启动时自动执行的(没有独立的"跑迁移"步骤),所以最危险的时刻就是新二进制第一次 start。

  ⚠️  两个关于脚本的提醒:
  1. manual_update.sh 不止更新 sub2api,还会 pull+rebuild+restart image-studio(它拉自己的 repo)。如果你这次只想动 sub2api,先确认 image-studio 的 repo 停在已知良好的
  commit,或干脆只手动执行脚本里 sub2api 那段,别整体跑。
  2. 脚本用 git pull --ff-only 拉 $HOME/app/sub2api 当前所在分支。确保那个分支已经包含 PR#39(合并 main 后部署,或把 repo 切到 PR 分支)。

  跑完后:
  sudo systemctl status sub2api          # 是否 active
  sudo journalctl -u sub2api -n 80       # 看启动日志,确认迁移全部 applied、无 error
  重点看日志里 165–173 是否都成功。冒烟测试关键路径(登录、网关转发、超级邀请页)。

  ---
  出错恢复

  第一层:只回滚代码(常见情况,秒级,不丢数据)

  适用于:新二进制能起但行为不对 / 迁移都成功了但业务逻辑有 bug。

  sudo systemctl stop sub2api
  sudo cp /opt/sub2api/sub2api.bak /opt/sub2api/sub2api   # 换回旧二进制
  sudo systemctl start sub2api
  旧二进制跑在已迁移的库上完全正常(前述增量结论)。新表闲置在那不影响。数据库不用动。

  第二层:连数据库一起还原(仅当 DB 真的坏了)

  适用于:迁移执行到一半失败、库进入不一致状态,或出现数据损坏。

  sudo systemctl stop sub2api                              # 断开所有连接
  PGPASSWORD=sub2api dropdb   -h localhost -U sub2api sub2api    # ★破坏性★
  PGPASSWORD=sub2api createdb -h localhost -U sub2api sub2api
  PGPASSWORD=sub2api pg_restore -h localhost -U sub2api -d sub2api \
    ~/backups/sub2api_<时间戳>.dump
  sudo cp /opt/sub2api/sub2api.bak /opt/sub2api/sub2api
  sudo systemctl start sub2api
  注意:dropdb 需要 sub2api 用户有建库权限/是 owner;若权限不足,改用 pg_restore --clean --if-exists -d sub2api ...(它逐对象 DROP 再建,不删整库)。

  一个边角:迁移 169 是 _notx

  169 是 CREATE INDEX CONCURRENTLY(非事务执行)。万一它中途失败,可能留下一个 invalid 索引——这不致命,手动 DROP INDEX CONCURRENTLY IF EXISTS <名> 清掉即可,重跑会重建。其余 8
  个都在各自事务里,失败自动回滚单条。

  Redis

  粘性会话/缓存在 Redis。回滚一般不用动它(缓存会自愈/过期)。若担心旧新结构串味,redis-cli FLUSHDB 清一下即可,通常不必。

  ---
  关于你的 pre-deploy 分支

  从 main 拉的备份分支没坏处,但代码层面其实不需要它——git 里 main 和 PR 分支都在,加上你已记录了
  deploy_prev_commit.txt,代码随时可复现。真正不可再生、必须备份的是数据库、运行中的二进制、config.yaml这三样,上面都覆盖了。
