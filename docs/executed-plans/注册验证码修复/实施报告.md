## 诊断

```bash
# 上传到服务器, 然后:
sudo bash diagnose_redis_timeout.sh \
  --duration 60 \
  --proc sub2api \
  --redis-port 6379 \
  --webproc <另一个 web 进程的名字>
```

把 `--webproc` 换成你实际的进程名(`nginx`/`caddy`/具体业务进程名). `pgrep -x` 找不到时会回退到 `pgrep -f`.

**关键: 在高峰期跑.** 平峰跑只能拿到上限/配置, 抓不到压力指标. 默认采样 30s, 推荐 `--duration 60` 覆盖一个完整波峰.

## 脚本会采集什么

只读, 不改任何东西. 12 大块, 都贴了"这是干嘛的":

1.  系统信息(内核/CPU/内存/uptime)
2.  **进程定位 + ulimit / FD 用量** — 你这台 2vCPU/4G, FD 上限非常关键
3.  **TCP 内核参数** — keepalive、backlog、临时端口、socket buffer、retries、conntrack
4.  **当前 ss 快照** — 总连接、TIME\_WAIT、Redis 连接、sub2api 拥有 socket 的状态分布
5.  **dmesg + journalctl** — conntrack 满表/OOM 留下的明确证据
6.  systemd unit 与 cgroup 限制(裸机也可能被 unit 限了)
7.  **PSI 压力指标** — `/proc/pressure/{cpu,memory,io}`, 看是哪类资源在压
8.  **Redis 服务端**: `INFO clients/stats/memory/commandstats`, `SLOWLOG`, `--latency`, 验证码/并发/计费 key 数量
9.  sub2api 进程的远端口分布 + Recv-Q/Send-Q 堆积情况
10.  **动态采样**: 每 5 秒打一行(TIME\_WAIT、established、Redis 连接、sub2api FD、CPU%), 共 N 次
11.  Redis CLIENT LIST 摘要 + idle 分位数
12.  输出文件路径

输出保存到 `/tmp/sub2api-diag-YYYYmmdd-HHMMSS.txt`.

## 关于你提到的"长连接 2-60s"

这点很重要, 我特意在脚本里加了:

*   第 4 块的 `ss -s` \+ sub2api socket 状态分布 — 看长连接占多少 FD/连接槽
*   第 9 块按对端端口分布 — 能直接看出长连接打到哪
*   第 10 块的动态采样 — 看高峰期 estab 是不是和 sub2api FD 数量相关性增长
*   `--webproc` 参数会单独统计另一个 web 进程的连接数, 用来估算它把多少 FD 吃掉了

**为什么这条线索很关键**: 2vCPU + 4G 是相当紧的配置, 如果另一个 web 进程在峰期同时维持几百到几千条 60s 长连接, 那就是:

*   两进程合计 FD 用量很容易撞到 ulimit 默认 1024
*   那些长连接每条都吃 socket buffer(默认配置每条 86KB rmem + 16KB wmem, 几百条就是几十 MB), 4G 内存里如果还跑着 PG 和 Redis, 内核 socket buffer 不够时会触发零窗口或丢包
*   长连接客户端如果是同步 keep-alive, 占着连接池等回包的同时, 给 sub2api 自己的 Redis 客户端贡献了 P 级别的 goroutine 抢占

## 脚本干什么
脚本输出会同时打到屏幕和保存到 `/tmp/sub2api-diag-YYYYmmdd-HHMMSS.txt`. 结构如下, 12 个分块, 每块都是 `========== N. 标题 ==========` 起头:

### 0\. 环境信息

日期、内核、系统版本、uptime、CPU 数、内存. 一两行.

### 1\. 定位 sub2api 进程

`pgrep` 找 PID, 然后 `ps -o pid,user,pcpu,pmem,rss,nlwp,etime,cmd`. 如果找不到会提示用 `--proc` 指定. 也会找 `--webproc` 那个进程.

### 2\. 文件描述符

*   `ulimit -Hn -Sn` (软/硬上限)
*   `fs.file-max`、`fs.nr_open`、`/proc/sys/fs/file-nr` (系统级总量)
*   进程的 `limits` (open files / processes)
*   **当前已用 FD 数** \+ FD 类型 top 20 分布 (socket/regular/anon\_inode 等)

> **要看的**: `current open FDs` 接近 `ulimit -Sn` 就是 FD 用完了.

### 3\. 内核网络参数

分四块打印:

*   TCP keepalive 三参数
*   backlog (`somaxconn`、`tcp_max_syn_backlog`、`syncookies`、`synack_retries`)
*   TIME\_WAIT (`ip_local_port_range`、`tw_reuse`、`fin_timeout`、`max_tw_buckets`)
*   socket buffer
*   TCP 重传次数
*   conntrack (max/count/buckets/timeout, 没装模块就显示 `unavailable`)

每块下面有一行 note 说"建议值".

### 4\. 当前 TCP 连接状态快照

*   `ss -s` 一行总览(estab/closed/synrecv/timewait 计数)
*   **Top 20 远端** (按 `host:port` 聚合 ESTABLISHED, 一眼能看出连接打到哪)
*   TIME\_WAIT 总数
*   Redis 端口的所有连接(最多 50 行) + 已建立连接数
*   sub2api 自己的 socket 状态分布(LISTEN/ESTAB/CLOSE\_WAIT/TIME\_WAIT 计数)

### 5\. 内核日志

*   最近 300 行 dmesg 里 grep `conntrack/too many/oom/killed/TCP:/drop`
*   journalctl 24h 内的 OOM

> **直接证据**: 出现 `nf_conntrack: table full` 或 `TCP: out of memory` 就是确诊.

### 6\. systemd / cgroup 限制

*   进程的 cgroup 路径
*   cgroup v2 的 `cpu.max`、`cpu.stat`(含 `nr_throttled`)、`memory.max`、`memory.current`、`memory.events`
*   如果是 systemd 启动的: `LimitNOFILE / CPUQuota / MemoryMax / TasksMax / Restart`

> 你裸机部署如果是 systemd unit 起的, 这块会显示 LimitNOFILE 实际值(很多发行版默认 1024, 极易踩雷).

### 7\. CPU/内存/IO 压力

*   `vmstat 1 5` (5 秒采样)
*   `/proc/pressure/{cpu,memory,io}` PSI 三组指标 (some/full × avg10/avg60/avg300)
*   `loadavg`

> **PSI avg10 持续 > 10%** 就是该资源在拖累请求.

### 8\. Redis 服务端

连接得上才会有内容, 否则跳过. 包括:

*   `INFO clients` (连接数、阻塞数、最大输入缓冲)
*   `INFO stats` 关键行 (`total_connections_received`、**`rejected_connections`**、`instantaneous_ops_per_sec`、`expired_keys`、`evicted_keys`、命中率)
*   `INFO memory` (used/peak/maxmemory/碎片率)
*   `INFO commandstats` 前 40 行 (哪个命令最耗时)
*   `CONFIG GET maxclients / timeout / tcp-keepalive / maxmemory-policy`
*   **SLOWLOG GET 20** (慢查询)
*   **`--latency`** 3 秒采样 + `--latency-history -i 1` 5 次
*   前缀 key 计数: `verify_code:* / concurrency:* / billing:* / rt:*`

> **要看的**: `rejected_connections` 不为 0 = Redis 自己拒过连接; `--latency` 平均超过 5ms 就异常.

### 9\. sub2api 进程网络细节

*   按对端口聚合的连接数 top 20 (能直接看出打到 Redis、PG、上游 AI、还是另一个 web)
*   **Recv-Q / Send-Q 非零的 socket** (有数据堆积就是 in-flight 卡住)
*   如果有 `--webproc`, 给那个进程也打一份: 总连接数 + 状态分布

### 10\. 动态采样 ${DURATION}s

**这是高峰期最有价值的一块**. 每 5 秒打一行:

```
[14:23:05] tw=412 estab=189 redis_estab=64 sub2api_fd=521 sub2api_cpu=78.3%
[14:23:10] tw=438 estab=203 redis_estab=64 sub2api_fd=534 sub2api_cpu=85.1%
[14:23:15] tw=455 estab=215 redis_estab=64 sub2api_fd=541 sub2api_cpu=92.7%
...
```

看连续几个时间点的数字变化:

*   `redis_estab` 顶到某个值后死活上不去 = Redis 连接池打满
*   `sub2api_fd` 单调上升不释放 = 长连接堆积
*   `sub2api_cpu` 持续 > 100/(2vCPU)% × 0.9 = CPU 见顶
*   `tw` 暴涨 = 大量短连接

### 11\. Redis CLIENT LIST 摘要

*   前 30 条 client (能看到每条的 addr/age/idle/cmd)
*   总连接数
*   **idle 分位数**: `count=N min=X p50=Y p95=Z max=W`

> p95 > 30s 而 max 几百秒, 说明池里很多连接闲置但没释放, 配合 #9 的 `tw` 就能判断是不是 keepalive 没生效.

### 12\. 输出位置

最后一行打印保存路径: `完整日志已保存到: /tmp/sub2api-diag-YYYYmmdd-HHMMSS.txt`.
