# 退费评估模块 — 实现方案 (Final)

## 背景

用户余额由多种来源构成（付费充值、赠金、管理员调整、推荐返佣），消费过程中各来源混合扣减。退费时管理员需要准确判断：某笔付费充值订单的额度已被消耗了多少、可退多少实付金额。

## 核心设计

独立只读评估模块，不修改现有退费执行流程。通过 FIFO（先进先出）算法将充值池总消耗按时间顺序分摊到各入账槽位，区分付费/免费来源，换算实付金额消耗。

---

## 数据源分类

| 来源 | 表 | FIFO 分类 | ratio |
|------|---|-----------|-------|
| 在线支付订单 | `payment_orders` (status IN completed/refunded/partially_refunded, order_type=balance) | 付费 | `pay_amount / amount`（支持1:2、1:3等倍率） |
| 兑换码充值 | `redeem_codes` (type=balance, value>0) | 付费 | 1.0（面值=实付） |
| 管理员调整(正值) | `redeem_codes` (type=admin_balance, value>0) | 免费 | 0 |
| 推荐人转余额 | `user_affiliate_ledger` (action=transfer, amount>0) | 免费 | 0 |
| 赠金 | `user_gifts` | 不参与FIFO | `gift_cost` 独立追踪 |

### 运营约束

- `redeem_codes` type=balance 统一算付费来源，运营层面约束不使用兑换码做免费促销。
- 兑换码 ratio 统一 1:1，不存在倍率放大情况。

## FIFO 分摊逻辑

### 总消耗计算

```
effective_used = Σ(usage_logs.recharge_cost)     -- API 消费
              + Σ(历史退费扣减的余额)              -- 退费消耗
```

退费扣减数据来源：`payment_audit_logs` WHERE action='REFUND_SUCCESS'，取 `detail.balanceDeducted`。

### 分摊算法

1. 收集所有入账槽位，按 credited_at ASC 排序
2. 将 `effective_used` 按 FIFO 依次分摊到各 slot：
   ```
   for slot in slots:
       consume = min(slot.amount, remaining)
       slot.consumed = consume
       slot.consumed_money = consume × slot.ratio
       slot.remaining = slot.amount - consume
       remaining -= consume
   ```
3. 对每笔付费订单，可退实付 = `pay_amount - consumed_money - refund_deducted × ratio`

### 退费记录处理

- 已退费订单仍出现在 FIFO 队列（status=refunded/partially_refunded）
- 退费扣减额计入 effective_used，确保 FIFO 正确反映"哪些额度被退费吃掉了"
- 管理员看到完整时间线，退费记录不会静默消失

### 边界情况

| 场景 | 处理 |
|------|------|
| 用户无使用记录 | effective_used=0，全额可退 |
| 管理员负值调整 | 不建槽位，效果体现在 current_pool 下降 |
| effective_used > 槽位总和 | consumed 到顶后停止（理论不应发生，意味着透支） |
| 多次退费同一用户 | current_pool 自然约束执行上限 |
| affiliate transfer 金额=0 | 不建槽位 |

## 退费后的联动（不联动）

| 项目 | 退费后是否变化 | 原因 |
|------|---|---|
| 赠金(user_gifts) | 不变 | `DeductFromRechargePool` 只减 balance，不动 gifts |
| 邀请返利 | 不变 | 退费流程无反向扣回逻辑 |
| 管理员调整 | 不变 | 退费只扣充值池 |
